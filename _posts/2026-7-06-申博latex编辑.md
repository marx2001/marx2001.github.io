## <center>说明</center>

本文用于记录我的**申博补充材料 PDF 自动化工作流**。这个工作流的目标是：将个人网页 `_posts` 文件夹中的 Markdown 内容，自动筛选、转换、排版，并生成一个可以和简历一起发送的正式 PDF 文档。

这个 PDF 不是网页截图，也不是简单的 Markdown 合并文件，而是一个面向申博使用的 **Research Portfolio / Supplementary Academic Portfolio**。它可以把研究笔记、原创脚本、代表性结果图、计算流程、项目记录等内容整理成一个正式的 PDF 附件。

目前工作流的核心特点如下：

- 直接读取个人网页 `_posts` 文件夹中的 Markdown 文件；
- 只筛选 `sci-note` 分类下的文章；
- 用 `pdf_order.txt` 控制哪些文章进入 PDF 以及文章顺序；
- 自动复制图片到纯英文路径，避免中文图片路径导致 LaTeX 编译失败；
- 自动压缩正文中过长的代码块；
- 完整代码自动收集到 PDF 最后的 Appendix；
- 支持单篇 Markdown 预览；
- 支持完整 PDF 构建；
- 支持强制全量重建；
- 支持 **PDF 专用 LaTeX 手动编辑层**，可以手动调整最终 PDF 中每篇文章的 `.tex` 排版。

---

## <center>整体逻辑</center>

整个流程可以理解为：

```text
个人网页 Markdown 文件
        ↓
build.py 自动读取并筛选 sci-note 分类
        ↓
按照 pdf_order.txt 决定 PDF 内容和顺序
        ↓
清理 front matter、处理图片路径、压缩长代码
        ↓
Pandoc 将 Markdown 转换成 converted/*.tex
        ↓
build.py 自动检查 pdf_tex_edits/ 中是否存在同名 .tex
        ↓
如果 pdf_tex_edits/ 中没有同名文件：
    自动复制 converted/*.tex 到 pdf_tex_edits/*.tex
如果 pdf_tex_edits/ 中已有同名文件：
    保留手动修改版，不覆盖
        ↓
最终 PDF 优先使用 pdf_tex_edits/*.tex
        ↓
main.tex 统一控制封面、目录、页眉页脚和整体格式
        ↓
XeLaTeX 编译生成最终 PDF
```

最终得到的文件是：

```text
output/Ruixiao_Ma_Research_Portfolio.pdf
```

这个文件就是可以和简历一起发送的申博补充材料 PDF。

## <center>PDF 专用 LaTeX 手动编辑层</center>

新增的 `pdf_tex_edits/` 文件夹用于解决一个关键问题：

```text
Markdown 自动转换出来的 LaTeX 有时版式不够理想；
但直接修改 converted/*.tex 又会在下次构建时被覆盖。
```

因此，现在工作流分成两层：

```text
converted/        自动生成层，不手动修改
pdf_tex_edits/    手动编辑层，专门用于调整 PDF 版式
```

核心规则非常简单：

```text
只要 pdf_tex_edits/ 中存在与 converted/ 同名的 .tex 文件，
最终 PDF 就优先使用 pdf_tex_edits/ 中的手动版本。
```

例如：

```text
converted/2026-2-5_471929fc.tex
pdf_tex_edits/2026-2-5_471929fc.tex
```

如果 `pdf_tex_edits/2026-2-5_471929fc.tex` 存在，最终 PDF 就会使用它，而不是 `converted/2026-2-5_471929fc.tex`。

## <center>第一次生成可手动编辑的 tex 文件</center>

不需要额外命令。正常运行构建命令即可：

```powershell
python build.py --only 投影能带
```

或者：

```powershell
python build.py
```

脚本会先生成：

```text
converted/xxx.tex
```

然后自动检查：

```text
pdf_tex_edits/xxx.tex
```

如果 `pdf_tex_edits/` 中没有同名文件，就会自动复制一份。之后你就可以打开 `pdf_tex_edits/` 中对应的 `.tex` 文件进行手动排版。

例如第一次运行后，可能会生成：

```text
pdf_tex_edits/2026-2-5_471929fc.tex
pdf_tex_edits/999_source_code_appendix.tex
```

其中：

```text
pdf_tex_edits/2026-2-5_471929fc.tex
```

用于调整正文文章排版。

```text
pdf_tex_edits/999_source_code_appendix.tex
```

用于调整最后完整代码附录的排版。

## <center>如何确认 PDF 使用的是手动 tex 文件</center>

可以打开：

```text
converted/includes.tex
```

如果里面是：

```tex
\input{pdf_tex_edits/2026-2-5_471929fc.tex}
```

说明最终 PDF 已经使用手动编辑层。

如果里面还是：

```tex
\input{converted/2026-2-5_471929fc.tex}
```

说明对应的手动 `.tex` 文件还没有生成，或者文件没有放在正确位置。

## <center>手动修改 tex 后如何重新生成 PDF</center>

手动修改：

```text
pdf_tex_edits/*.tex
```

之后，不需要加 `--force`，直接运行：

```powershell
python build.py --only 投影能带
```

或者生成完整版本：

```powershell
python build.py
```

即可。

`--force` 主要用于重新清理缓存和自动生成文件。如果只是修改 `pdf_tex_edits/*.tex`，一般不需要 `--force`。

## <center>什么时候改 Markdown，什么时候改 tex</center>

推荐按下面原则维护：

```text
长期内容修改：
    改个人网页 _posts/*.md

PDF 局部排版修改：
    改 pdf_tex_edits/*.tex

PDF 全局格式修改：
    改 main.tex

文章选择和顺序：
    改 pdf_order.txt

脚本逻辑：
    改 build.py
```

不要直接改：

```text
converted/*.tex
_tmp_md/*.md
```

因为它们属于自动生成的中间文件。

## <center>适合在 pdf_tex_edits 中手动调整的内容</center>

`pdf_tex_edits/*.tex` 适合处理 Markdown 自动转换后不够美观的局部排版问题，例如：

- 修改标题层级；
- 删除不适合 PDF 的网页内容；
- 调整图片大小；
- 把普通图片改成标准 `figure` 环境；
- 调整图片前后空白；
- 手动加入 `\clearpage`；
- 调整表格宽度；
- 压缩代码块字号；
- 删除正文中冗长的代码预览；
- 优化单篇文章的说明结构。

例如，Pandoc 有时会把网页中的标题转换得不够理想，如果出现类似：

```tex
\#\#

说明
```

可以手动改成：

```tex
\subsection{说明}
```

如果正文代码块太大，可以改成：

```tex
\begingroup
\small
\begin{verbatim}
...
\end{verbatim}
\endgroup
```

如果还嫌大，可以把 `\small` 换成：

```tex
\footnotesize
```

或者：

```tex
\scriptsize
```

## <center>如何重新生成某个手动 tex 初稿</center>

如果你已经手动修改过某个 `pdf_tex_edits/*.tex`，脚本默认不会覆盖它。

如果想放弃手动修改，重新从 Pandoc 自动转换结果开始排版，可以删除对应文件：

```text
pdf_tex_edits/2026-2-5_471929fc.tex
```

然后重新运行：

```powershell
python build.py --only 投影能带
```

脚本会再次从：

```text
converted/2026-2-5_471929fc.tex
```

复制一份新的初稿到：

```text
pdf_tex_edits/2026-2-5_471929fc.tex
```


```text
_pdf_images/img_90456cd7_bg-t4.png
```

这样可以避免中文路径导致的编译问题。

## <center>Markdown 文件的 front matter 要求</center>

想被纳入 PDF 的 Markdown 文件，必须属于 `sci-note` 分类。推荐 front matter 写法如下：

```yaml
---
layout: post
title: "(原创脚本)绘制精美的投影能带图"
subtitle: "Experience Sharing"
background: '/img/bg-sci-note.jpg'
categories: sci-note
permalink: /sci-note_posts/20260205-fatband
---
```

其中最关键的是：

```yaml
categories: sci-note
```

如果一个 Markdown 文件没有这个分类，即使它在 `_posts` 文件夹中，也不会被纳入申博 PDF 的候选范围。

## <center>pdf_order.txt 的作用</center>

`pdf_order.txt` 用来控制：

1. 哪些 Markdown 文件进入最终 PDF；
2. 这些 Markdown 文件在 PDF 中的排列顺序。

示例：

```text
# Only files under category/categories: sci-note will be included.
# One Markdown filename per line.
# Lines beginning with # are ignored.

2026-2-5-绘制精美的投影能带图.md
2026-3-10-另一个科研脚本.md
2026-4-1-某个计算流程记录.md
```

注意，这里写的是 `_posts` 文件夹中的**原始 Markdown 文件名**，不是转换后的 `.tex` 文件名。

## <center>标准工作流</center>

### 第一步：找到需要加入 PDF 的 Markdown 文件

首先在个人网页 `_posts` 文件夹中找到目标文件，例如：

```text
2026-2-5-绘制精美的投影能带图.md
```

确认它的 front matter 中有：

```yaml
categories: sci-note
```

如果没有，需要先添加或修改分类。

---

### 第二步：把文件名加入 pdf_order.txt

打开 `pdf_order.txt`，新增一行：

```text
2026-2-5-绘制精美的投影能带图.md
```

如果想调整 PDF 中的章节顺序，只需要调整 `pdf_order.txt` 中各行文件名的顺序。

---

### 第三步：预览单篇文章

如果只是想检查新加入的这篇文章在 PDF 中的效果，可以运行：

```powershell
python build.py --only 2026-2-5
```

或者用文件名中的关键词：

```powershell
python build.py --only 投影能带
```

这会生成预览 PDF：

```text
output/Ruixiao_Ma_Research_Portfolio_PREVIEW.pdf
```

预览 PDF 只包含匹配到的 Markdown 文件，适合快速检查单篇内容、图片和代码压缩效果。

第一次预览时，脚本也会自动生成对应的：

```text
pdf_tex_edits/*.tex
```

---

### 第四步：手动调整 PDF 版式

如果预览 PDF 中存在标题、图片、代码块、分页、空白等问题，就打开对应的：

```text
pdf_tex_edits/*.tex
```

进行手动修改。

修改完成后，再运行：

```powershell
python build.py --only 投影能带
```

不需要加 `--force`。

---

### 第五步：生成最终完整 PDF

确认单篇预览没有问题后，运行：

```powershell
python build.py
```

这会根据 `pdf_order.txt` 生成完整 PDF：

```text
output/Ruixiao_Ma_Research_Portfolio.pdf
```

这个 PDF 才是最终的申博补充材料版本。

---

### 第六步：必要时强制全量重建

如果修改了以下内容：

- `build.py`
- `main.tex`
- 代码压缩规则
- 图片路径处理规则
- LaTeX 模板
- 目录结构
- 缓存文件出现异常

建议运行：

```powershell
python build.py --force
```

这个命令会清理缓存和自动生成文件，并重新构建所有内容。

注意：`--force` 不会删除 `pdf_tex_edits/` 中已经存在的手动 `.tex` 文件，因此已经完成的手动排版调整不会被误删。

需要特别注意：

```powershell

```

只生成预览 PDF，输出文件是：


它只包含匹配到的 Markdown 文件。


才会根据 `pdf_order.txt` 生成完整总 PDF，输出文件是：


```text

        ↓
必要时修改 pdf_tex_edits/*.tex
        ↓
再次 python build.py --only 关键词
        ↓
满意后 python build.py
        ↓
检查完整 PDF
```

## <center>长代码块的处理规则</center>

为了避免完整代码严重影响正文阅读，当前工作流会自动处理长代码块。

规则如下：

```text
超过 80 行的代码块：
正文中只显示前 30 行；
完整代码自动放入 Appendix: Source Codes。
```

例如正文中会显示：

```text
Code block shortened for readability.
Language: python. Full length: 425 lines.
The first 30 lines are shown below.
The complete code is collected in Appendix: Source Codes.
```

完整代码会出现在 PDF 最后的：

```text
Appendix: Source Codes
```

这样可以同时满足两个需求：

1. 正文保持可读性；
2. 完整代码仍然保留在 PDF 中。

如果正文中的代码预览仍然太长，可以在 `pdf_tex_edits/*.tex` 中手动删除或压缩正文预览，同时保留 Appendix 中的完整代码。

## <center>图片处理规则</center>

个人网页中图片可能写成：

```markdown
![figure](/img/某个中文文件夹/bg-t4.png)
```

或者：

```markdown
<img src="/img/某个中文文件夹/bg-t4.png">
```

`build.py` 会自动：

1. 识别图片路径；
2. 在原始 `img` 文件夹中找到图片；
3. 复制到 `_pdf_images/`；
4. 重命名为纯英文文件名；
5. 将 Markdown 中的图片路径改写为新路径。

例如：

```text
/img/某个中文文件夹/bg-t4.png
```

会被转换为：

```text
_pdf_images/img_90456cd7_bg-t4.png
```

这样可以保证 LaTeX 编译时不直接处理中文图片路径。

如果想手动调整 PDF 中图片大小，可以在 `pdf_tex_edits/*.tex` 中修改，例如：

```tex
\includegraphics[width=0.85\linewidth]{_pdf_images/img_90456cd7_bg-t4.png}
```

## <center>常见问题</center>

### 问题 1：新 Markdown 文件没有进入 PDF

检查三点：

1. 该文件是否在 `_posts` 文件夹中；
2. front matter 是否有 `categories: sci-note`；
3. 文件名是否已经加入 `pdf_order.txt`。

---

### 问题 2：预览命令找不到新文件

如果使用：

```powershell
python build.py --only 新文件关键词
```

却没有选中文件，通常是因为该文件还没有加入 `pdf_order.txt`。

当前逻辑下，如果存在 `pdf_order.txt`，`--only` 会在 `pdf_order.txt` 已列出的文件中查找。因此标准做法是：

```text
先把文件名加入 pdf_order.txt
再运行 python build.py --only 关键词
```

---

### 问题 3：pdf_tex_edits 中没有对应 tex 文件

先运行一次：

```powershell
python build.py --only 关键词
```

或者：

```powershell
python build.py --force
```

然后检查：

```text
pdf_tex_edits/
```

如果仍然没有，打开：

```text
converted/includes.tex
```

查看是否已经输入了 `pdf_tex_edits/*.tex`。正常情况下应类似：

```tex
\input{pdf_tex_edits/2026-2-5_471929fc.tex}
```

---

### 问题 4：修改了 pdf_tex_edits 但 PDF 没变化

检查：

```text
converted/includes.tex
```

确认里面输入的是：

```tex
\input{pdf_tex_edits/xxx.tex}
```

而不是：

```tex
\input{converted/xxx.tex}
```

另外确认你修改的是项目目录中的：

```text
pdf_tex_edits/xxx.tex
```

而不是下载目录或其他位置的同名文件。

---

### 问题 5：图片找不到

检查 Markdown 中图片路径是否确实指向个人网页的 `img` 文件夹。

常见可识别写法包括：

```markdown
![figure](/img/文件夹名/bg-xxx.png)
```

```markdown
![figure](img/文件夹名/bg-xxx.png)
```

```html
<img src="/img/文件夹名/bg-xxx.png">
```

如果图片只有文件名，例如：

```markdown
![figure](bg-t4.png)
```

脚本会在 `SOURCE_IMG_DIR` 中递归搜索该文件名。

---

### 问题 6：修改了图片但 PDF 没更新

如果只是替换了图片，但 Markdown 文件没有变化，缓存可能不会重新处理该文件。

解决方法：

```powershell
python build.py --force
```

或者轻微修改并保存对应 Markdown 文件，然后重新运行：

```powershell
python build.py
```

---

### 问题 7：LaTeX 编译出现代码高亮相关报错

如果出现类似：

```text
Undefined control sequence
Environment Shaded undefined
Environment Highlighting undefined
```

说明旧的 converted 文件或 Pandoc 高亮命令没有清理干净。

解决方法：

```powershell
python build.py --force
```

## <center>推荐的日常使用顺序</center>

最常用的完整流程如下：

```text
1. 在个人网页中新建或修改 Markdown 文件；
2. 确保 front matter 中包含 categories: sci-note；
3. 将该 Markdown 文件名加入 pdf_order.txt；
4. 运行 python build.py --only 关键词，检查单篇预览；
5. 如果是第一次构建，检查 pdf_tex_edits/ 中是否已生成同名 tex；
6. 修改 Markdown 内容或图片；
7. 如果只是 PDF 版式问题，则修改 pdf_tex_edits/*.tex；
8. 再次运行 python build.py --only 关键词；
9. 满意后运行 python build.py；
10. 检查完整 PDF；
11. 将 output/Ruixiao_Ma_Research_Portfolio.pdf 作为申博补充材料发送。
```

## <center>备注</center>

这个工作流适合长期维护。以后新增科研笔记、脚本说明、研究项目记录时，只需要维护个人网页中的 Markdown 文件和 `pdf_order.txt`；如果 PDF 自动排版效果不理想，再通过 `pdf_tex_edits/*.tex` 做局部修整即可。

本质上，这个流程实现了：

```text
网页内容长期维护一次
PDF 正文自动生成
PDF 局部版式单独手动微调
```

这样既能保证个人网页和申博补充材料之间保持一致，又能避免 PDF 被 Markdown 自动转换结果限制，从而兼顾自动化和最终排版质量。
