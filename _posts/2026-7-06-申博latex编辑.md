## <center>说明</center>

本文用于记录我的**申博补充材料 PDF 自动化工作流**。这个工作流的目标是：将个人网页 `_posts` 文件夹中的 Markdown 内容，自动筛选、转换、排版并生成一个可以和简历一起发送的 PDF 文档。

这个 PDF 并不是普通网页截图，也不是简单的 Markdown 合并文件，而是一个面向申博使用的 **Research Portfolio / Supplementary Academic Portfolio**。它可以把研究笔记、原创脚本、代表性结果图、计算流程、项目记录等内容整理成一个正式的 PDF 附件。

目前工作流的核心特点如下：

- 直接读取个人网页中的 Markdown 文件；
- 只筛选 `sci-note` 分类下的文章；
- 用 `pdf_order.txt` 控制哪些文章进入 PDF 以及文章顺序；
- 自动复制图片到纯英文路径，避免中文图片路径导致 LaTeX 编译失败；
- 自动压缩正文中过长的代码块；
- 完整代码自动收集到 PDF 最后的 Appendix；
- 支持单篇 Markdown 预览；
- 支持完整 PDF 构建；
- 支持强制全量重建；
- 支持 PDF 专用 LaTeX 手动编辑层，可以直接手动调整最终 PDF 使用的 `.tex` 片段。

---

## <center>整体逻辑</center>

整个流程可以理解为：

```text
个人网页 Markdown 文件
        ↓
build.py 自动读取并筛选 sci-note 分类
        ↓
按照 pdf_order.txt 决定 PDF 内容和顺序
        ↓
清理 front matter、处理图片路径、压缩长代码
        ↓
Pandoc 将 Markdown 转换成 LaTeX 片段
        ↓
如果存在 pdf_tex_edits/ 同名 .tex，则优先使用手动编辑版本
        ↓
main.tex 统一控制封面、目录、页眉页脚和整体格式
        ↓
XeLaTeX 编译生成最终 PDF
```

最终得到的文件是：

```text
output/Ruixiao_Ma_Research_Portfolio.pdf
```

这个文件就是可以和简历一起发送的申博补充材料 PDF。

_tmp_md/
converted/

## <center>PDF 专用 LaTeX 手动编辑层</center>

新增的 `pdf_tex_edits/` 文件夹用于解决一个重要问题：

```text
Markdown 自动转换出来的 LaTeX 有时版式不够理想，
但直接修改 converted/*.tex 又会在下次构建时被覆盖。
```

因此，现在工作流被分成两层：

```text
converted/       自动生成层，不建议手动编辑
pdf_tex_edits/   手动编辑层，专门用于调整 PDF 版式
```

核心规则是：

```text
如果 pdf_tex_edits/ 中存在与 converted/ 同名的 .tex 文件，
build.py 会优先使用 pdf_tex_edits/ 中的手动版本。
如果不存在，则继续使用 converted/ 中的自动版本。
```

也就是说，真正应该手动调整的是：

```text
pdf_tex_edits/*.tex
```

而不是：

```text
converted/*.tex
```

---

### 第一次创建手动 tex 编辑文件

例如想手动调整“投影能带”这篇文章的 PDF 版式，可以先运行：

```powershell
python build.py --only 投影能带 --init-tex-edit
```

这个命令会先按原流程生成 `converted/*.tex`，然后把对应的 `.tex` 复制到：

```text
pdf_tex_edits/
```

之后你就可以打开 `pdf_tex_edits/` 中对应的 `.tex` 文件，手动修改 LaTeX 内容，例如：

- 调整图片宽度；
- 调整图片前后的空白；
- 手动加入 `\clearpage`；
- 修改标题层级；
- 删除不适合 PDF 的网页内容；
- 把图片改成 `figure` 环境；
- 手动调整表格、代码块或分页位置。

---

### 查看 Markdown 和 tex 文件的对应关系

由于转换后的 `.tex` 文件名会带有稳定 hash，不建议靠肉眼猜。可以运行：

```powershell
python build.py --list-tex-map
```

或者只查看某一篇：

```powershell
python build.py --only 投影能带 --list-tex-map
```

脚本会输出类似信息：

```text
[1] 2026-2-5-绘制精美的投影能带图.md
    auto tex   : converted/2026-2-5_8a1b2c3d.tex
    manual tex : pdf_tex_edits/2026-2-5_8a1b2c3d.tex
    currently used: pdf_tex_edits/2026-2-5_8a1b2c3d.tex
```

同时会在下面生成一个索引文件：

```text
pdf_tex_edits/_README_tex_edit_map.txt
```

以后不知道该改哪个 `.tex` 时，先看这个文件即可。

---

### 手动修改 tex 后如何重新生成 PDF

手动修改 `pdf_tex_edits/*.tex` 后，直接运行：

```powershell
python build.py --only 投影能带
```

或者生成完整版本：

```powershell
python build.py
```

只要 `pdf_tex_edits/` 中存在对应的 `.tex` 文件，脚本就会自动优先使用它。

---

### 临时忽略手动 tex 编辑层

如果怀疑手动改坏了 `.tex`，可以临时忽略手动层，使用自动转换结果编译：

```powershell
python build.py --only 投影能带 --ignore-tex-edits
```

如果这样可以正常编译，而不加 `--ignore-tex-edits` 时失败，说明问题基本就在 `pdf_tex_edits/` 中手动修改过的 `.tex` 文件里。

---

### 重新覆盖手动 tex 文件

一般不建议使用这个命令，因为它会覆盖已经手动修改过的内容：

```powershell
python build.py --only 投影能带 --init-tex-edit --overwrite-tex-edit
```

它适合在以下情况使用：

- 原始 Markdown 内容大幅更新；
- 想放弃之前的手动 LaTeX 调整；
- 希望重新从 Pandoc 自动转换结果开始排版。

---

### 推荐的 tex 手动编辑原则

建议遵循下面的优先级：

```text
内容长期维护：优先改个人网页 Markdown
PDF 局部排版：改 pdf_tex_edits/*.tex
全局格式风格：改 main.tex
文章顺序选择：改 pdf_order.txt
```

不要直接改：

```text
converted/*.tex
_tmp_md/*.md
```

因为这些仍然属于自动生成的中间文件。


## <center>个人网页 Markdown 文件位置</center>

个人网页文章原始位置为：

```text
C:\Users\Administrator\Documents\GitHub\marx2001.github.io\_posts
```

图片根目录为：

```text
C:\Users\Administrator\Documents\GitHub\marx2001.github.io\img
```

由于 `img` 文件夹内部存在中文命名的子文件夹，LaTeX 直接读取这些路径时可能不稳定。因此，`build.py` 会自动将被引用的图片复制到：

```text
_pdf_images/
```

并重命名为纯英文路径，例如：

```text
_pdf_images/img_90456cd7_bg-t4.png
```

这样可以避免中文路径导致的编译问题。

## <center>Markdown 文件的 front matter 要求</center>

想被纳入 PDF 的 Markdown 文件，必须属于 `sci-note` 分类。推荐 front matter 写法如下：

```yaml
---
layout: post
title: "(原创脚本)绘制精美的投影能带图"
subtitle: "Experience Sharing"
background: '/img/bg-sci-note.jpg'
categories: sci-note
permalink: /sci-note_posts/20260205-fatband
---
```

其中最关键的是：

```yaml
categories: sci-note
```

如果一个 Markdown 文件没有这个分类，即使它在 `_posts` 文件夹中，也不会被纳入申博 PDF 的候选范围。

---

## <center>pdf_order.txt 的作用</center>

`pdf_order.txt` 用来控制：

1. 哪些 Markdown 文件进入最终 PDF；
2. 这些 Markdown 文件在 PDF 中的排列顺序。

示例：

```text
# Only files under category/categories: sci-note will be included.
# One Markdown filename per line.
# Lines beginning with # are ignored.

2026-2-5-绘制精美的投影能带图.md
2026-3-10-另一个科研脚本.md
2026-4-1-某个计算流程记录.md
```

注意，这里写的是 `_posts` 文件夹中的**原始 Markdown 文件名**，不是转换后的 `.tex` 文件名。

## <center>标准工作流</center>

### 第一步：找到需要加入 PDF 的 Markdown 文件

首先在个人网页 `_posts` 文件夹中找到目标文件，例如：

```text
2026-2-5-绘制精美的投影能带图.md
```

确认它的 front matter 中有：

```yaml
categories: sci-note
```

如果没有，需要先添加或修改分类。

---

### 第二步：把文件名加入 pdf_order.txt

打开 `pdf_order.txt`，新增一行：

```text
2026-2-5-绘制精美的投影能带图.md
```

如果想调整 PDF 中的章节顺序，只需要调整 `pdf_order.txt` 中各行文件名的顺序。

---

### 第三步：预览单个 Markdown 文件

如果只是想检查新加入的这篇文章在 PDF 中的效果，可以运行：

```powershell
python build.py --only 2026-2-5
```

或者用文件名中的关键词：

```powershell
python build.py --only 投影能带
```

这会生成预览 PDF：

```text
output/Ruixiao_Ma_Research_Portfolio_PREVIEW.pdf
```

预览 PDF 只包含匹配到的 Markdown 文件，适合快速检查单篇内容、图片和代码压缩效果。

---

### 第四步：生成最终完整 PDF

确认单篇预览没有问题后，运行：

```powershell
python build.py
```

这会根据 `pdf_order.txt` 生成完整 PDF：

```text
output/Ruixiao_Ma_Research_Portfolio.pdf
```

这个 PDF 才是最终的申博补充材料版本。

---

### 第五步：强制全量重建

如果修改了以下内容：

- `build.py`
- `main.tex`
- 代码压缩规则
- 图片路径处理规则
- LaTeX 模板
- 目录结构
- 缓存文件出现异常

建议运行：

```powershell
python build.py --force
```

这个命令会清理缓存和自动生成文件，并重新构建所有内容。`--force` 不会删除 `pdf_tex_edits/` 中的手动 `.tex` 文件，因此已经完成的手动排版调整不会被强制重建误删。

## <center>关于预览和总 PDF 的区别</center>

需要特别注意：

```powershell
python build.py --only 关键词
```

只生成预览 PDF，输出文件是：

```text
Ruixiao_Ma_Research_Portfolio_PREVIEW.pdf
```

它只包含匹配到的 Markdown 文件。

而：

```powershell
python build.py
```

才会根据 `pdf_order.txt` 生成完整总 PDF，输出文件是：

```text
Ruixiao_Ma_Research_Portfolio.pdf
```

因此，标准流程应该是：

```text
新增 md 到 pdf_order.txt
        ↓
python build.py --only 关键词
        ↓
检查单篇预览效果
        ↓
python build.py
        ↓
检查完整 PDF
```

## <center>长代码块的处理规则</center>

为了避免完整代码严重影响正文阅读，当前工作流会自动处理长代码块。

规则如下：

```text
超过 80 行的代码块：
正文中只显示前 30 行；
完整代码自动放入 Appendix: Source Codes。
```

例如正文中会显示：

```text
Code block shortened for readability.
Language: python. Full length: 425 lines.
The first 30 lines are shown below.
The complete code is collected in Appendix: Source Codes.
```

完整代码会出现在 PDF 最后的：

```text
Appendix: Source Codes
```

这样可以同时满足两个需求：

1. 正文保持可读性；
2. 完整代码仍然保留在 PDF 中。

## <center>图片处理规则</center>

个人网页中图片可能写成：

```markdown
![figure](/img/某个中文文件夹/bg-t4.png)
```

或者：

```markdown
<img src="/img/某个中文文件夹/bg-t4.png">
```

`build.py` 会自动：

1. 识别图片路径；
2. 在原始 `img` 文件夹中找到图片；
3. 复制到 `_pdf_images/`；
4. 重命名为纯英文文件名；
5. 将 Markdown 中的图片路径改写为新路径。

例如：

```text
/img/某个中文文件夹/bg-t4.png
```

会被转换为：

```text
_pdf_images/img_90456cd7_bg-t4.png
```

这样可以保证 LaTeX 编译时不直接处理中文图片路径。

## <center>常见问题</center>

### 问题 1：新 Markdown 文件没有进入 PDF

检查三点：

1. 该文件是否在 `_posts` 文件夹中；
2. front matter 是否有 `categories: sci-note`；
3. 文件名是否已经加入 `pdf_order.txt`。

---

### 问题 2：预览命令找不到新文件

如果使用：

```powershell
python build.py --only 新文件关键词
```

却没有选中文件，通常是因为该文件还没有加入 `pdf_order.txt`。

当前逻辑下，如果存在 `pdf_order.txt`，`--only` 会在 `pdf_order.txt` 已列出的文件中查找。因此标准做法是：

```text
先把文件名加入 pdf_order.txt
再运行 python build.py --only 关键词
```

---

### 问题 3：图片找不到

检查 Markdown 中图片路径是否确实指向个人网页的 `img` 文件夹。

常见可识别写法包括：

```markdown
![figure](/img/文件夹名/bg-xxx.png)
```

```markdown
![figure](img/文件夹名/bg-xxx.png)
```

```html
<img src="/img/文件夹名/bg-xxx.png">
```

如果图片只有文件名，例如：

```markdown
![figure](bg-t4.png)
```

脚本会在 `SOURCE_IMG_DIR` 中递归搜索该文件名。

---

### 问题 4：修改了图片但 PDF 没更新

如果只是替换了图片，但 Markdown 文件没有变化，缓存可能不会重新处理该文件。

解决方法：

```powershell
python build.py --force
```

或者轻微修改并保存对应 Markdown 文件，然后重新运行：

```powershell
python build.py
```

---

### 问题 5：LaTeX 编译出现代码高亮相关报错

如果出现类似：

```text
Undefined control sequence
Environment Shaded undefined
Environment Highlighting undefined
```

说明旧的 converted 文件或 Pandoc 高亮命令没有清理干净。

解决方法：

```powershell
python build.py --force
```

当前版本已经关闭 Pandoc 代码高亮，并在 `main.tex` 中加入了兼容定义。

- [ ] 文件名是否为 `Ruixiao_Ma_Research_Portfolio.pdf`；
- [ ] PDF 是否可以独立打开，不依赖本地图片路径。

## <center>推荐的日常使用顺序</center>

最常用的完整流程如下：

```text
1. 在个人网页中新建或修改 Markdown 文件；
2. 确保 front matter 中包含 categories: sci-note；
3. 将该 Markdown 文件名加入 pdf_order.txt；
4. 运行 python build.py --only 关键词，检查单篇预览；
5. 修改 Markdown 内容或图片；
6. 再次预览；
7. 满意后运行 python build.py；
8. 检查完整 PDF；
9. 将 output/Ruixiao_Ma_Research_Portfolio.pdf 作为申博补充材料发送。
```

## <center>备注</center>

这个工作流适合长期维护。以后新增科研笔记、脚本说明、研究项目记录时，只需要维护个人网页中的 Markdown 文件和 `pdf_order.txt`，不需要手动复制内容到 LaTeX 中。

本质上，这个流程实现了：

```text
网页内容维护一次
PDF 申博材料自动生成
```

这样可以保证个人网页和申博补充材料之间保持一致，同时显著减少重复排版工作。
