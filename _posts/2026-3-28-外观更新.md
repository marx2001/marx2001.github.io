这篇文章记录了博客升级搜索功能和代码复制功能的过程，界面更清晰美观，更加专业，以及更加方便。
两类更新：

1. 如何把 **Jekyll 站内搜索** 从“能用”改成“标题 + 正文尽量不漏召回”
2. 如何把 **代码块 + 复制按钮** 做成更稳定、更好维护的版本

如果你也在用 Jekyll、GitHub Pages 或类似的静态博客，这篇内容基本可以直接参考。

---

## 一、这次更新要解决什么问题

最开始遇到的不是单一问题，而是一串互相关联的小问题：

### 1. 搜索功能不够稳

已有搜索功能虽然能用，但存在几个明显短板：

- 搜索标题和正文时容易漏结果
- 中文、多词组合、特殊写法不够稳
- 文章多了以后，搜索开始变慢
- 技术文章里的大段代码会污染搜索索引

### 2. 代码块和复制按钮体验一般

代码块部分也有多个问题：

- 代码块样式偏呆板
- 复制按钮视觉不统一
- 代码块一开始出现了双层灰背景
- 部分按钮存在白色残留背景
- 点击复制后，对勾状态不生效
- 滚动代码时，按钮会跟着内容一起跑，体验不好

这次更新的核心，不是单纯“改个颜色”，而是把这两块都梳理成更清晰、可维护的方案。

## 二、搜索功能：目标不是复杂，而是“尽量不漏”

这次搜索优化一开始就明确了目标：

> 不追求复杂筛选，不追求推荐系统，只追求“标题 + 正文尽量不漏召回”。

这句话很重要，因为它决定了后面的技术路线。

如果目标是：

- 搜索尽量稳
- 尽量少漏文章
- 适合静态博客

那么继续使用 **前端静态搜索** 是合理的，不需要为了这个需求专门上后端搜索服务。

---

## 三、原有搜索方案的判断

原本博客的搜索方案属于典型的静态站点前端搜索：

- 页面：`search.html`
- 索引：`search.json`
- 检索逻辑：`search.js`
- 底层依赖：`lunr.js`

这种方案的优点很明显：

- 部署简单
- GitHub Pages 友好
- 不依赖服务器
- 小博客足够轻量

但它的缺点也明显：

- 直接 `idx.search(q)` 的检索方式过于朴素
- 一旦文章多起来，`search.json` 会越来越大
- 技术博客里的代码块会干扰正文搜索

---

## 四、搜索功能是怎么改的

### 1. 查询逻辑不再只搜整句

如果用户输入一个较长的短语，或者中英文混合关键词，直接整句搜索很容易漏。

因此搜索逻辑后来改成了更宽松的模式：

- 整句搜索
- 拆词搜索
- 前缀匹配
- 原始字符串包含兜底

这里最关键的是最后一层：

### 2. 加入“原始字符串包含”兜底

这是这次搜索升级里最实用的一步。

思路很简单：

- 标题里直接包含关键词 → 强召回
- 正文里直接包含关键词 → 兜底召回

它不一定最优雅，但对“尽量不漏文章”的目标非常有效。

### 3. 标题命中优先

在排序上，标题命中的文章会优先排到前面，这样搜索结果更符合日常直觉。

---

## 五、搜索性能为什么会变差

当文章数量慢慢变多后，真正的性能瓶颈并不在页面本身，而在两个地方：

### 1. `search.json` 越来越大

原始做法通常是把整篇文章内容都放进去，例如：

```liquid
"content": {{ post.content | strip_html | strip_newlines | jsonify }}
```

这样做的后果是：

- 文件体积越来越大
- 首次加载搜索页就很重
- 前端建索引时间越来越长

### 2. 页面一打开就立刻加载索引

这意味着用户即使只是打开搜索页，还没开始搜索，也要先下载并构建整份索引。

---

## 六、搜索性能是怎么优化的

### 1. 正文改为截断，不再塞整篇全文

例如改成：

```liquid
"content": {{ post.content | strip_html | strip_newlines | truncate: 1200 | jsonify }}
```

这样做的意义是：

- 索引变小
- 首次加载更快
- 对大多数文章来说，前几段正文已经足够支持搜索

### 2. 改成延迟加载索引

搜索逻辑后来改成：

- 页面打开时不立刻建索引
- 用户开始输入后，再加载 `search.json`
- 只构建一次，后续复用

这种方式对于文章较多的博客非常有效。

### 3. 控制结果数

例如限制：

```javascript
MAX_RESULTS = 50
```

因为搜索结果不需要一次渲染几百条，限制显示数量可以减少页面渲染负担。

---

## 七、技术博客里的代码块，为什么会污染搜索

这个问题是这次搜索优化里一个很关键的发现。

技术博客的文章经常包含大段代码，而在 Jekyll 里，代码块经过渲染后依然会进入 `post.content`。

这意味着：

- 代码内容会进入 `search.json`
- 代码字符会占用正文截断长度
- 如果文章前面代码很多，真正的正文就会被截掉

结果就是：

- 搜索结果变差
- 搜索更容易命中代码而不是正文
- 有些文章明明正文里有关键词，却因为正文没进索引而搜不到

---

## 八、如何在 `search.json` 里跳过代码块

正确做法不是在前端再删除代码，而是在 **生成 `search.json` 的时候就先删掉代码块**。

思路是：

1. 先把 `<pre>...</pre>` 对应内容从文章 HTML 中剔除
2. 再 `strip_html`
3. 再做 `truncate`

这样做的好处非常直接：

- 代码不进入搜索索引
- 截断长度留给真正正文
- 搜索质量明显更好

对于技术博客来说，这是非常值得做的一步。

---

## 九、代码块样式部分：为什么主要改 `main.scss`

代码块视觉和复制按钮的样式问题，本质上首先是 CSS 问题。

所以一开始的判断是正确的：

> 大部分视觉问题优先改 `main.scss`，不要一开始就怀疑 HTML 结构。

后续的实践也证明了这一点：

- 背景色
- 圆角
- 按钮视觉
- tooltip
- 按钮位置
- 图标比例

这些主要都靠 `main.scss` 完成。

## 十、复制按钮的类名是怎么确认的

通过开发者工具排查后，确认页面里的按钮类名是：

```text
.copy-btn
```

这个确认很重要，因为如果类名都没对上，后面所有样式覆盖都容易失效。

此外还发现，旧按钮曾经带有一套内联样式，这也是后来产生冲突的根源之一。

---

## 十一、双层灰背景是怎么出现的

一开始代码块样式改动后，页面上出现了“双层灰背景”：

- 外层容器一层灰
- 内层 `pre` 又一层灰

最后发现，正确的职责划分应该是：

### 外层容器
只负责：

- 定位
- 容纳按钮

### `pre`
负责：

- 真正的代码块背景
- 圆角
- 边框
- 阴影

因此后来做法变成：

- 外层背景透明
- 只有 `pre` 保留那层浅灰背景

这样视觉上才干净。

---

## 十二、复制按钮为什么一开始会有白边残留

这个问题一开始看起来像单纯的 CSS 边框问题，但实际更复杂。

原因不是只有一个，而是几种情况叠加：

- 页面里存在不同来源的按钮
- 有的按钮是旧脚本生成的
- 有的按钮被额外容器包裹
- 有的按钮还带旧的 inline style

这说明：

> 当同一个功能由两套脚本或两套结构同时控制时，CSS 问题会变得非常难判断。

所以真正的解决方法，不是一直补选择器，而是要先消除旧逻辑冲突。

---

## 十三、为什么要把按钮改成图标化

原来的“复制代码”纯文字按钮虽然可用，但有几个问题：

- 占空间
- 风格不够简洁
- 容易和代码块整体视觉不协调

因此后来改成了：

- 默认显示复制图标
- 鼠标悬停时显示 tooltip“复制”
- 点击成功后图标短暂变成对勾

这个交互形式更紧凑，也更符合现在常见的代码块工具按钮风格。

---

## 十四、为什么最后选择“纯 CSS 画图标”

中间一度尝试过：

- 内嵌 SVG
- data URI 图标

但在静态博客环境下，这种方式容易遇到：

- SCSS 编译转义问题
- 引号 / 特殊字符问题
- data URI 在不同环境下不稳定

最终改成：

> 直接用纯 CSS 画复制图标和对勾。

这样虽然实现上略微原始，但非常稳：

- 不依赖外部图片
- 不依赖 SVG 文件
- 不受资源路径影响
- 更容易微调比例

---

## 十五、为什么“复制成功后变对勾”一开始总是不成功

这部分是本次更新中最典型的排错案例。

### 第一步误判：以为是 CSS 对勾没画对

一开始大家很自然会想到：

- 对勾位置错了
- `.copied::before` 没覆盖干净
- 默认图标还残留

这些判断不完全错，但都不是根因。

### 第二步发现：点击后 class 根本没变

继续排查后发现：

- 点击后按钮并没有短暂变成 `copy-btn copied`

这说明真正的问题不是“图标画错”，而是：

> JS 根本没给按钮加 `.copied` 类。

### 第三步发现：页面里有旧复制脚本

再继续查，发现 `head.html` 里原来存在一段旧的内联脚本。它会：

- 自己创建 `.copy-btn`
- 自己写按钮样式
- 点击后直接把文字改成“已复制”

这样一来，新脚本虽然存在，但用户实际点到的按钮并不一定受新逻辑控制。

### 第四步发现：旧脚本删掉后按钮直接消失

删掉旧脚本后，按钮完全消失，于是又怀疑“是不是按钮根本没创建”。

最后通过 Network 查到：

> `/assets/js/copy-code.js` 是 404。

这说明问题其实更基础：

- 新脚本根本没加载到页面
- 那当然不会创建按钮
- 也不会加 `.copied`

这个案例非常典型：

> 表面看起来像 CSS 问题，实际是旧脚本冲突 + 新脚本路径错误。

---

## 十六、复制按钮脚本最后是怎么修好的

### 1. 修正文件路径

最终确认 `copy-code.js` 必须放在：

```text
assets/js/copy-code.js
```

然后在文章页布局中加载：

```html
<script src="{{ '/assets/js/copy-code.js' | relative_url }}"></script>
```

修正路径后，脚本成功执行。

### 2. 删除旧脚本

`head.html` 中旧的内联复制脚本彻底删除，只保留一套复制逻辑。

这一步非常重要，否则：

- 样式会互相打架
- 点击行为会互相干扰
- 很难判断当前按钮到底由谁生成

### 3. 新脚本负责完整流程

新的 `copy-code.js` 负责：

- 遍历所有 `pre`
- 创建 `.copy-btn`
- 绑定点击复制事件
- 复制成功后添加 `.copied`
- 1 秒后移除 `.copied`

这样 JS 和 CSS 的职责才对应起来。

---

## 十七、为什么复制按钮会随着代码滚动而消失

这是后期又出现的一个交互问题。

原因很直接：

- 按钮一开始是 `appendChild` 到 `pre` 里面的
- 而 `pre` 正好是滚动容器
- 所以滚动代码时，按钮也会跟着代码内容一起移动

这就导致用户滚到代码中下部时，右上角复制按钮不见了。

---

## 十八、如何让按钮始终固定在代码块右上角

最终做法是：

- 不再 `pre.appendChild(button)`
- 改成找到外层 wrapper：
  - `.highlighter-rouge`
  - `.highlight`
  - `figure.highlight`
- 把按钮 append 到 wrapper 上

这样就形成了更合理的结构：

### `pre`
负责：
- 滚动代码内容

### wrapper
负责：
- 定位复制按钮

这样无论代码怎么滚，按钮都固定在外层右上角，不再消失。

---

## 十九、为什么后来改成“图标与圆心绑定”

如果按钮图标靠写死坐标定位，例如：

- `top: 12px`
- `left: 12px`

那后面一旦改按钮大小或位置，就要一起重调图标，维护非常麻烦。

因此后来改成：

- 图标 `left: 50%`
- 图标 `top: 50%`
- 再通过 `transform: translate(...)` 细调

这样做之后：

- 调按钮位置 → 只改 `.copy-btn` 的 `top/right`
- 调按钮大小 → 只改 `.copy-btn` 的 `width/height`
- 图标自动跟着圆心走

这是一种非常值得保留的写法。

## 二十、这次更新后各文件的职责

### `main.scss`
负责：

- 代码块视觉
- 行内代码样式
- 滚动条样式
- 复制按钮视觉
- tooltip 样式
- `.copied` 对勾状态

### `assets/js/copy-code.js`
负责：

- 查找代码块
- 创建复制按钮
- 绑定复制事件
- 控制 `.copied` 状态
- 确保按钮插入到正确外层容器

### `post.html`
负责：

- 在文章页底部加载 `copy-code.js`

### `head.html`
当前只保留：

- highlight.js 资源
- 基础页面资源

旧的复制按钮脚本已经删除。

### `search.js`
负责：

- 标题 + 正文宽松召回
- 整句 / 拆词 / 前缀 / 兜底匹配

### `search.json`
负责：

- 生成搜索索引
- 跳过代码块
- 截断正文长度
- 控制索引体积

## 二十一、这次更新里最值得记住的经验

### 1. 不要同时保留两套同功能脚本

复制按钮的问题之所以会越来越乱，根本原因就是旧脚本和新脚本同时存在。

经验很简单：

> 一个功能，只保留一套逻辑来源。

### 2. 资源是否成功加载，必须优先确认

这次 `copy-code.js` 404 非常典型。

如果没先查 Network，很容易在 CSS 和 DOM 上浪费大量时间。

经验顺序应该是：

1. 先看 Network
2. 再看 Console
3. 再查 DOM
4. 最后才是 CSS

### 3. 技术博客的搜索索引不能无脑塞全文

技术文章里代码很多，如果全文直接进索引：

- 会拖慢搜索
- 会污染召回结果
- 会挤掉正文有效信息

因此删除代码块后再生成索引，是非常值得做的一步。

### 4. 图标与圆心绑定后，维护成本明显降低

这是本次样式调整中很有价值的一个结果。

后续无论继续调比例、位置还是按钮大小，都会轻松很多。

## 二十四、最终结论

这次更新表面上看是两件小事：

- 改搜索
- 改复制按钮

但实际涉及的内容很完整：

- 搜索逻辑
- 索引体积控制
- 技术博客中的代码块搜索污染问题
- CSS 视觉层次整理
- JS 事件绑定与状态切换
- 静态资源路径排查
- 页面布局中的脚本注入方式

最终最大的收获不是某一段代码，而是一套更清晰的维护方法：

> **功能逻辑尽量单一来源，样式结构尽量职责分离，问题排查按照“资源加载 → JS 行为 → DOM 结构 → CSS 视觉”逐层推进。**

这套思路，后面再改博客其他功能时也会继续有用。
