## 3. 移动端问题排查过程

## 3.1 首轮判断：不是 viewport 缺失，而是导航栏和样式层冲突

最开始排查移动端异常时，先看了：
- `head.html`
- `navbar.html`
- `home.html`
- `autopage_category.html`
- `main.scss`
- `_navbar.scss`

### 结论
`head.html` 中 viewport 已经存在，因此“整站完全不响应式”并不是因为忘写：

```html
<meta name="viewport" content="width=device-width, initial-scale=1, shrink-to-fit=no">
```

真正的问题主要集中在两点：

1. **导航栏移动端折叠菜单逻辑被自定义 flex 样式破坏**
2. **首页 / 分类页文章列表样式写在 `#posts-container` 下，但模板最初并没有这个容器，导致一部分响应式样式根本没命中**

---

## 3.2 `#posts-container` 问题

在 `main.scss` 中，首页和分类页很多样式都绑定在：

```scss
#posts-container { ... }
```

但最开始的 `home.html` 和 `autopage_category.html` 文章列表结构没有统一包在 `#posts-container` 下。

### 结果
- 某些首页 / 分类页的移动端字号、间距、分页样式没有真正生效
- 这不是唯一 bug，但属于一类“样式写了却没命中模板”的问题

### 后续修正
首页和分类页模板都改成了带有：

```html
<div id="posts-container">
  ...
</div>
```

的结构。

---

## 3.3 移动端 menu 按钮“收回一瞬间又弹开”的问题

这是本阶段最典型的导航栏 bug。

### 现象
- 点击 `Menu`
- 导航似乎先收起
- 但马上又重新下拉，继续遮挡标题和背景图

### 根因判断
问题不是 JS 失效，而是 **Bootstrap 的 collapse 逻辑被自定义 CSS 顶掉**。

核心原因包括：
- `.navbar-collapse` 在移动端也被强行写成 `display: flex`
- 导致 `.collapse.show` / `.collapse:not(.show)` 的默认显示隐藏逻辑失效
- 结果就是按钮点击时，JS 改变了 class，但 CSS 又把它硬显示出来

### 对应修正思路
后续改动集中在：
- `_navbar.scss`
- `navbar.html`

主要原则是：
- **移动端不再强制 `.navbar-collapse` 使用 flex**
- **桌面端再启用 flex 布局**
- `Menu` 按钮放到右上角，而不是居中

---

## 3.4 导航栏结构与桌面端居中效果的平衡

中间曾尝试过“标准 Bootstrap 结构优先”的方案，这能解决移动端折叠问题，但会改变 PC 端导航居中视觉。

### 经验结论
- 单纯用标准 Bootstrap 结构，功能上更稳，但会改变原来桌面端的居中感
- 如果想兼顾：
  - **PC 端：继续保持整体居中**
  - **移动端：Menu 固定右上角，正常折叠**
  
  就需要分开写桌面端和移动端样式，而不是共用一套布局策略

### 最终判断
在实际部署中，之前某一版 `_navbar.scss` 已经足够可用，因此后续没有继续大动 `_navbar.scss`，而是把重点转向分页和模板统一。

## 4. 分页（上一页 / 下一页）问题的排查与修复

这一部分是本阶段讨论最多、也最容易反复的内容。

---

## 4.1 首页 / 分类页分页最开始的问题

主要有几类：

1. PC 端分页按钮黏在一起，不够美观
2. 按钮上方出现双线
3. 移动端按钮上下排布过长，比例不好看
4. 中文 / 英文混用，带箭头，不统一
5. 首页、分类页、文章内部的分页逻辑和样式不统一

---

## 4.2 首页与分类页分页结构的统一

为了让 `main.scss` 中的分页样式能真正命中，后续统一采用类似这样的结构：

```html
<div id="pagination">
  {% if paginator.previous_page %}
    <a class="btn btn-primary pagination-btn pagination-prev" ...>上一页</a>
  {% else %}
    <span class="pagination-placeholder"></span>
  {% endif %}

  {% if paginator.next_page %}
    <a class="btn btn-primary pagination-btn pagination-next" ...>下一页</a>
  {% else %}
    <span class="pagination-placeholder"></span>
  {% endif %}
</div>
```

### 这样做的原因
原始模板里分页通常还是：
- `clearfix`
- `float-left`
- `float-right`

而后面写的 CSS 已经转向：
- `.pagination-btn`
- `.pagination-prev`
- `.pagination-next`
- `.pagination-placeholder`

如果 HTML 不跟着统一，后面继续补 CSS 往往“看起来没效果”。

---

## 4.3 PC 端分页按钮间距和左右分布

### 初始问题
PC 端“上一页 / 下一页”按钮：
- 间距不够大
- 看起来像粘在一起
- 不能很好地和内容区宽度对齐

### 试过的方案
1. 改按钮宽度
2. `flex + justify-content: space-between`
3. `margin-left: auto` / `margin-right: auto`
4. 单独推右按钮

### 经验结论
- 仅改按钮宽度不能真正解决“对称分布”的问题
- 如果 HTML 结构仍是旧的 `float-left / float-right`，后面很多 flex 规则并不会达到预期
- 真正要稳定，需要：
  - 容器结构统一
  - 分页按钮类名统一
  - 尽量避免 `clearfix` 伪元素继续干扰 flex

---

## 4.4 `clearfix` 对分页布局的干扰

后期通过现象判断出：

```html
<div id="pagination" class="clearfix">
```

这种写法在使用 flex 布局时可能引入伪元素 `::before / ::after`，干扰分页按钮的对齐。

### 对应经验
如果分页容器本身已经用 flex，`clearfix` 一般没必要继续保留。  
中间也尝试通过 CSS 关闭：

```scss
#pagination::before,
#pagination::after {
  content: none !important;
  display: none !important;
}
```

来抑制这类干扰。

---

## 4.5 中文化与去箭头

后续统一要求：

- 首页分页：`上一页` / `下一页`
- 分类页分页：`上一页` / `下一页`
- 文章内部上一篇下一篇：`上一篇` / `下一篇`
- 去掉左右箭头

### 涉及文件
- `home.html`
- `autopage_category.html`
- `post.html`

### 经验结论
分页文字问题不要试图只用 CSS 解决，必须回到模板里直接改文本。

---

## 4.6 移动端分页问题

### 现象
移动端曾出现：
- 上一页和下一页上下堆叠
- 按钮太长，像横条
- 和 PC 端风格不一致

### 原因
`main.scss` 中旧的移动端分页规则把它写成了：

```scss
flex-direction: column;
width: 100%;
max-width: 220px;
```

### 经验结论
如果想让移动端也尽量接近 PC 端风格，应避免这类“强行纵向堆叠 + 拉满宽度”的旧规则。  
但由于样式中重复的 `#pagination` 规则很多，单纯在末尾追加一段覆盖规则，有时效果并不稳定。

因此后面反复强调的一点是：

> 仅追加 CSS 不一定能解决问题，必须先保证模板结构和目标样式是同一套体系。

---

## 5. 双线问题的排查过程

这部分是本阶段最典型的“必须靠开发者工具定位来源”的问题。

---

## 5.1 最初判断失误：以为两根线都来自分页

一开始尝试通过：
- 去掉 `#pagination` 的 `border-top`
- 去掉最后一个 `<hr>`
- 调整 `top: -1px`

等方法处理，但效果不稳定，说明两根线来源并不单一。

---

## 5.2 开发者工具定位结果

后续通过开发者工具逐步确认：

### （1）footer 前确实存在单独的 `<hr>`
这会在页面更下方形成一根线。

### （2）分页区域本身存在 `border-top`
这也是一根线。

### （3）分类页模板中，文章循环后仍然**无条件输出 `<hr>`**
也就是说，最后一篇文章后面也会有一根线。

最终就出现了“三个来源可能混在一起”的情况：

1. 文章列表最后一个 `<hr>`
2. `#pagination` 的 `border-top`
3. footer 前单独的 `<hr>`

---

## 5.3 经验结论：要先分清是哪一根线

开发者工具最终告诉我们的核心经验是：

- **footer 的 `<hr>` 是底部那根线**
- **分页按钮上方那组线，不是 footer 的线**
- 分页按钮上方如果出现双线，通常是：
  - 最后一篇文章后的 `<hr>`
  - 加上 `#pagination` 的 `border-top`

因此处理方式不是“乱删 CSS”，而是：

- 如果要保留分页自己的线  
  → 就应该让最后一篇文章后不再输出 `<hr>`
- footer 前那根线要不要保留  
  → 是另一个独立决定

---

## 5.4 分类页最终正确改法

`autopage_category.html` 中原来是：

```liquid
{% for post in paginator.posts %}
  ...
  <hr>
{% endfor %}
```

最终应改成：

```liquid
{% for post in paginator.posts %}
  ...
  {% unless forloop.last %}
    <hr>
  {% endunless %}
{% endfor %}
```

### 经验意义
这说明很多“看起来像 CSS 问题”的视觉 bug，其实根源是模板中多输出了结构性元素。

---

## 6. 首页与分类页不一致的问题

后期一个重要发现是：

- `home.html`
- `autopage_category.html`

虽然看起来类似，但细节上常常不一致：
- 有的有 `#pagination`
- 有的没有
- 有的文章循环后用 `unless forloop.last`
- 有的还是无条件 `<hr>`
- 有的分页按钮类名统一了
- 有的还残留旧类

### 经验结论
首页和分类页如果想统一体验，不能只“看起来差不多”，而要把：
- 容器
- 循环中的 `<hr>`
- 分页按钮类名
- 分页文字

全部对齐。

---

## 7. 本阶段试错经验总结

## 7.1 仅在 `main.scss` 末尾叠加覆盖规则，不一定可靠
原因包括：
- 旧规则重复太多
- 模板结构没跟着改
- 旧类名与新类名不匹配
- 伪元素或 float 逻辑仍在干扰

---

## 7.2 先确认“是 CSS 命中问题，还是模板结构问题”
这次很多问题最终都证明：
- 不是 SCSS 语法不对
- 而是 HTML 模板结构没跟着改

典型例子：
- `#posts-container` 没加，导致样式没命中
- 分页按钮没用 `pagination-next` / `pagination-prev` 类，导致后加规则没用
- 分类页仍无条件输出 `<hr>`，导致双线一直存在

---

## 7.3 用浏览器开发者工具比猜 CSS 更有效
尤其是这几类问题：
- 哪根线来自哪里
- 当前页面到底命中了哪个规则
- `footer` 的 `<hr>` 还是 `pagination` 的 `border-top`
- 选中的元素到底是 `hr`、`div#pagination` 还是伪元素

这是本阶段一个非常重要的经验。

---

## 7.4 不要一次性大改所有文件
对导航栏、分页、首页、分类页、文章页这种多模板问题，最好：
- 一次只修一个场景
- 一次只验证一个页面
- 先定位来源，再改模板或样式

否则会出现“这里修好了，那里又坏了”的反复情况。

---

## 8. 当前阶段的稳定结论

截至这一阶段，已经比较明确的结论有：

1. **移动端导航栏的主问题不是 viewport，而是 collapse 与自定义 flex 冲突**
2. **`#posts-container` 是否存在，直接影响首页 / 分类页样式是否命中**
3. **分页问题不能只靠 CSS 叠加，必须统一 HTML 结构**
4. **分页按钮中文化与去箭头要在模板中改**
5. **分类页分页上方双线的核心来源，是最后一篇文章后无条件输出 `<hr>`**
6. **footer 前的 `<hr>` 是另一根独立横线，不应与分页双线混为一谈**
7. **开发者工具是定位这类视觉问题的关键工具**

---

## 9. 建议后续维护方式

后面如果继续维护这个博客，建议：

### 9.1 统一模板规范
尤其是：
- 首页
- 分类页
- 文章页

把分页区、文章循环、`<hr>` 输出逻辑都统一。

### 9.2 `main.scss` 分区管理
把分页相关样式集中到一个单独区域，不要在文件不同位置重复写多个 `#pagination`。

### 9.3 每次修改后立刻用 devtools 验证
重点看：
- HTML 结构是否真变了
- 当前命中的 CSS 规则是哪一条
- 是否是缓存问题

### 9.4 避免“补丁式叠加”
当样式重复太多时，继续加覆盖规则只会越来越乱。  
与其补丁，不如收敛成一份明确的最终结构。

---

## 10. 本阶段最有价值的经验性信息

如果只提炼最关键的经验，有以下几点：

1. **Jekyll 页面显示问题，很多时候不是 SCSS 写错，而是模板结构和样式目标不一致**
2. **导航栏、分页、分隔线这类问题，CSS 和模板必须配套修改**
3. **看到多条线时，不要先删 CSS，要先用开发者工具确认每一根线的真实来源**
4. **移动端与 PC 端可以共用主题，但不应强行共用完全相同的布局机制**
5. **调试时先定位“当前页面到底用了哪个模板、命中了哪条规则”，比盲改更重要**
