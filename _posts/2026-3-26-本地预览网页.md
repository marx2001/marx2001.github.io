# Windows 下安装 Ruby 与本地预览 Jekyll 的流程总结

## 一、这部分内容的目的

这次整理的目标，是把本对话中关于以下内容的操作流程完整记录下来：

1. 如何在 Windows 上安装 Ruby。
2. 如何安装 Bundler 和 Jekyll。
3. 如何在本地运行 Jekyll 站点并预览网页效果。
4. 如何理解本地预览时常见的终端输出、警告和错误。
5. 本次实际遇到过的问题以及对应的解决方法。

这份记录适合作为以后重新配置电脑、本地重装环境、或者帮助别人搭建同类博客时的参考文档。

---

## 二、为什么最开始会报错

最开始在项目目录里执行：

```bat
bundle install
```

终端报错：

```text
'bundle' 不是内部或外部命令，也不是可运行的程序或批处理文件。
```

### 这说明什么

这通常说明以下几种情况之一：

1. Ruby 还没有安装。
2. Bundler 还没有安装。
3. Ruby 已经安装，但没有加入系统 PATH。
4. 命令行窗口没有重新打开，环境变量还没有刷新。

也就是说，这不是博客代码本身的问题，而是本地 Ruby 环境还没有准备好。

## 三、Windows 下安装 Ruby 的流程

## 1. 安装 RubyInstaller for Windows

在 Windows 上，推荐安装：

**RubyInstaller for Windows**

建议选择：

- 64 位版本
- 带 Devkit 的版本（Ruby+Devkit）

### 安装时要注意的点

安装过程中要注意：

1. 勾选 **Add Ruby executables to your PATH**
2. 安装完成后继续执行 **ridk install**
3. 安装 MSYS2 / Devkit

### 为什么要装 Devkit / MSYS2

因为很多 Ruby gems 在 Windows 上都需要依赖它，尤其是 Jekyll 项目在安装部分依赖时，经常会需要 Devkit 环境。

---

## 2. 安装完成后看到的 Ruby 终端菜单

安装完成后，打开 Ruby 终端时，会看到类似几个选项：

- `irb:>`  
- `Core and stdlib API`
- `gem doc server`
- `C:\> ruby enabled command line`
- `Install MSYS2-Devkit necessary for many gems`

### 这几个选项里，实际最重要的是两个

#### 第一个：进入命令行环境

应当选择：

```text
C:\> ruby enabled command line
```

这是后面执行 Ruby、gem、bundle、jekyll 命令的入口。

#### 第二个：安装 Devkit

还应该执行：

```text
Install MSYS2-Devkit necessary for many gems
```

这个步骤建议完成，不要跳过。

---

## 四、安装完成后先验证环境

进入 Ruby 命令行后，先执行：

```bat
ruby -v
gem -v
```

### 这两条命令的作用

- `ruby -v`：检查 Ruby 是否安装成功
- `gem -v`：检查 RubyGems 是否可用

如果这两条命令都能正常输出版本号，说明 Ruby 基础环境已经可以用了。

---

## 五、安装 Bundler 和 Jekyll

在 Ruby 命令行中执行：

```bat
gem install bundler jekyll
```

### 这一步的作用

安装：

- Bundler
- Jekyll

安装完成后，再检查：

```bat
bundle -v
jekyll -v
```

### 如果这里能正常输出版本号

就说明：

- Bundler 安装成功
- Jekyll 安装成功
- 本地已经具备运行 Jekyll 项目的基础能力

---

## 六、进入项目目录

在 Windows 下，项目目录在不同盘符时，推荐使用：

```bat
cd /d D:\aa\1\marx2001.github.io
```

### 为什么要加 `/d`

因为 `/d` 可以让命令同时切换盘符和目录。  
如果只写 `cd D:\aa\1\marx2001.github.io`，有时不会真正切换到对应盘符。

---

## 七、安装项目依赖

进入博客项目目录后，执行：

```bat
bundle install
```

### 这一步的作用

根据项目根目录里的：

```text
Gemfile
```

安装这个项目真正需要的所有 Ruby 依赖。

这一步非常重要，因为即使系统已经装好了 Jekyll，本项目可能还依赖其他 gems，例如：

- `jekyll-feed`
- `jekyll-sitemap`
- `jekyll-paginate-v2`

---

## 八、本地预览 Jekyll 网站

## 1. 启动本地服务

在项目根目录执行：

```bat
bundle exec jekyll serve
```

如果想提升实时刷新体验，也可以执行：

```bat
bundle exec jekyll serve --livereload
```

### 这两条命令的作用

- `bundle exec jekyll serve`：启动本地 Jekyll 服务器
- `--livereload`：保存文件后浏览器更容易自动刷新

---

## 2. 启动成功时的典型输出

如果启动成功，终端通常会出现类似：

```text
Server address: http://127.0.0.1:4000/
Server running... press ctrl-c to stop.
```

### 这说明什么

这说明：

- Jekyll 已经成功构建
- 本地服务器已经启动
- 现在可以在浏览器打开站点预览效果

---

## 3. 浏览器里打开本地地址

一般打开以下地址之一即可：

```text
http://127.0.0.1:4000/
```

或者：

```text
http://localhost:4000/
```

如果首页能打开，说明本地预览成功。

---

## 九、本地修改后如何查看效果

## 1. 普通页面内容、布局、样式修改

如果修改的是以下内容，通常不需要完全重启服务，只需要：

- 保存文件
- 刷新浏览器

常见包括：

- 文章内容
- 普通页面 `.md` / `.html`
- `_includes/`
- `_layouts/`
- 大多数 CSS / SCSS
- 导航栏内容
- 页面正文文本

---

## 2. 哪些情况建议重启 `jekyll serve`

如果修改的是下面这些，建议停止服务后重新运行：

- `_config.yml`
- `Gemfile`
- 插件配置
- 分页逻辑
- 自动分类页逻辑
- 新增 layout / 新增 plugin
- 新增或大改生成规则

### 原因

这些内容往往会影响 Jekyll 的构建流程本身，单纯刷新浏览器不一定会重新读取这些配置。

---

## 3. 正确重启方式

在当前终端按：

```text
Ctrl + C
```

停止当前服务，然后再执行：

```bat
bundle exec jekyll serve
```

如果改过 `Gemfile`，则先执行：

```bat
bundle install
```

再执行：

```bat
bundle exec jekyll serve
```

## 十一、这次实际遇到过的问题

## 问题 1：`bundle` 不是内部或外部命令

### 表现

执行：

```bat
bundle install
```

时报错，提示系统找不到 `bundle`。

### 原因

通常是：

- Ruby 未安装
- Bundler 未安装
- Ruby 没有加入 PATH
- 命令行没有重新打开

### 解决方法

1. 安装 RubyInstaller for Windows
2. 勾选加入 PATH
3. 安装 MSYS2-Devkit
4. 执行：

```bat
gem install bundler jekyll
```

5. 重新打开命令行

---

## 问题 2：不知道 Ruby 终端菜单里该选哪个

### 表现

安装 Ruby 后打开界面，不清楚应该选：

- `irb`
- `gem doc`
- 还是其他选项

### 解决方法

优先选择：

```text
C:\> ruby enabled command line
```

后续如果还没装 Devkit，再选择：

```text
Install MSYS2-Devkit necessary for many gems
```

---

## 问题 3：本地服务起来后，不确定是否正常

### 现象

终端里出现一大段输出和 warning，不确定有没有启动成功。

### 判断标准

最重要的是有没有出现：

```text
Server address: http://127.0.0.1:4000/
Server running... press ctrl-c to stop.
```

只要出现了这类输出，就说明：

- 站点已经成功在本地启动
- 浏览器可以访问

---

## 问题 4：本地有大量 warning，不知道是否要马上处理

### 现象

例如出现一堆 Sass deprecation warning。

### 结论

如果本地服务器已经成功运行，这类 warning 一般不会立刻阻止预览。  
它们往往是：

- 主题老旧
- Bootstrap / SCSS 写法过时

### 处理建议

如果只是 warning，而不是 build failed，可以先继续做功能修改。  
后续有空再集中处理样式层面的 warning。

---

## 问题 5：本地正常，线上不正常

这次对话中，本地预览成功后，后来还出现过：

- 本地分类页正常
- GitHub Pages 上仍然 404

### 这说明什么

说明：

- 本地预览成功 ≠ 线上部署一定成功
- 本地环境只是验证“构建逻辑是否工作”
- 线上还要看 GitHub Actions 是否跑对了、是否部署了最新产物、文章日期是否影响发布等

### 经验

本地预览解决的是“本地构建问题”，  
GitHub 上显示异常还要继续排查：

- workflow 是否触发
- artifact 是否包含页面
- 未来日期文章是否被排除
- Pages Source 是否正确

## 十三、后续维护建议

1. 如果换电脑或重装系统，优先先恢复 Ruby / Bundler / Jekyll 环境。
2. 每次大改 `_config.yml`、插件、分页逻辑后，最好重启一次 `jekyll serve`。
3. 本地预览只负责验证构建与页面效果，不代表 GitHub 线上发布一定同步正确。
4. 如果以后再安装依赖失败，先检查：
   - Ruby 是否在 PATH
   - `bundle -v` 是否可用
   - `Gemfile` 是否已经更新
5. 如果站点越来越复杂，建议保持：
   - 本地先测试
   - 再 push
   - 最后再看 GitHub Actions 与线上页面

---

## 十四、最终总结

这次关于 Ruby 安装与 Jekyll 本地预览的流程，最终明确了以下几点：

1. Windows 下必须先有正确的 Ruby 环境，才能运行 Bundler 和 Jekyll。
2. `bundle install` 报错，优先排查 Ruby / Bundler 环境，而不是先怀疑项目代码。
3. 本地预览的核心命令是：

```bat
bundle exec jekyll serve
```

4. 成功启动的关键标志是看到本地服务器地址，例如：

```text
http://127.0.0.1:4000/
```

5. 普通页面内容修改通常不需要重启，但配置、插件、Gemfile 修改建议重启。
6. 本地能跑通，是后续排查 GitHub Pages 问题的前提。

这部分流程虽然属于环境准备，但它实际上决定了后续所有博客修改、分页调试、页面优化是否能够高效进行。
