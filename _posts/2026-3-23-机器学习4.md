当前对话工作总结说明（ML 拓扑代理模型项目）。

### 项目背景

当前项目围绕 8 参数最小模型（磁群 `{123,342}`、z 向共线自旋假设）构建机器学习代理模型，目标是：输入一组参数，快速判断该参数点是否为拓扑候选，并进一步服务于参数空间搜索、相图分析和后验 Chern 验证。

### 本次对话的核心工作主线

本次对话主要围绕“第一个应用：相图代理模型 / 拓扑候选筛选器”展开，并在后续延伸到：

1. 机器学习模型的用途讨论（3 个应用方向）
2. 预测脚本、joint 规则脚本、测试集生成脚本
3. Mathematica / WL 导出 `hr.dat` 脚本

5. 阈值扫描脚本

## 一、模型的三个主要应用（当前建议的论文主线）

1. **应用一：相图代理模型 / 拓扑候选快速筛选**
   - 输入一组 8 参数
   - 输出模型判断：是否绝缘、是否拓扑、是否可靠
   - 适合作为“高精度拓扑候选预筛器”
   - 这是当前对话中实现最完整、最推荐优先推进的应用

2. **应用二：参数敏感性 / 控制轴识别**
   - 通过 `feature importance / SHAP / 局部扫描` 分析
   - 识别哪些参数或组合变量最主导拓扑相界
   - 当前结论：`t2` 和 `r3` 是拓扑相界主轴
   - 与解析模型中 `e、t、r` 的物理关系相互印证

3. **应用三：8 参数拟合 / 反演辅助**
   - 从真实材料低能带或 Wannier 结果出发，拟合等效 8 参数
   - 再把拟合得到的参数点放入 ML 相图中
   - 这是连接真实材料的关键增强方向
   - 当前对话中讨论了理论方案，但尚未正式展开完整脚本链条

## 二、当前阶段的主要结论

1. 训练好的三层模型已经能正常工作，且当前预测脚本已修好 22 维特征输入问题：
   - 原始输入只需 8 个参数：  
     `e1, e2, t1, t2, r1, r2, r3, r4`
   - 脚本内部自动补齐训练时用到的 14 个组合特征：  
     `e0, m, rbar, d12, d34, dpair, A13, A24, A14, A23, B1, B2, B3, B4`

2. 组合特征的定义来自：
   - `05_B_next_stage_refine_debug_fixed.py`
   - 核心函数：`add_signed_features(df)`

3. 三层模型当前使用目录（已统一到 `scripts` 相对路径口径）：
   - `05_ml_models/levelA_insulator_strict/xgboost/model.joblib`
   - `05_ml_models/me_final_boundary_top20_refine/xgboost/model.joblib`
   - `05_ml_models/levelC_reliability_strict_v2/xgboost/model.joblib`

4. 现阶段最推荐的用途不是“完全替代 Chern 计算”，而是：
   - 高 `precision` 的 topo candidate screening
   - 即：先用 ML 报出高置信候选，再做后验 Chern 验证

5. 在严格 consistent `sample_id` 链条下，100 个样本的最终验证结果为：

   - `precision = 0.8461538461538461`

## 三、关于“究竟该提升 accuracy 还是 precision”的结论

用户当前的核心目标是：

> 只关心模型说“这组参数对应的是拓扑”这句话准不准。

因此最重要的指标是：

- `Precision`

### 原因

- `Precision = TP / (TP + FP)`
- 它直接回答：模型一旦报 topo，这个判断有多大概率是真的
- 这和后续 expensive 的 `band / Chern` 计算成本最直接相关

### 为什么 Accuracy 不是当前第一优先

- `Accuracy` 会受到非 topo 样本占比影响
- 即便模型对 trivial 判得很好，accuracy 也可能高
- 但这不等价于“报 topo 是否真的准”

因此：

- 当前优先目标：提升 `Precision`
- 其次再考虑 `Accuracy`
- `Recall` 不是当前主目标（除非想尽量少漏 topo）

## 五、重要的 WL / Chern / compare 链条说明

1. 一开始出现“只匹配上 152/200”的问题，原因不是模型不准，而是：
   - 预测 CSV、WL 导出、Chern 汇总没有共享同一套 `sample_id`
   - 导致 compare 脚本只能退回到参数 `round` 匹配

2. 最终通过 consistent 版本解决：
   - 抽样 CSV 中保留 `sample_id`
   - WL 导出时把同一个 `sample_id` 写入 `hr.dat` 文件头
   - Chern 脚本从 `hr.dat` 文件头读出 `sample_id`
   - compare 脚本按 `sample_id` 严格 1:1 对齐

3. Chern 脚本是否统一：
   - 逻辑上已经统一，因为 `read_hr_file()` 会读取 `# sample_id`
   - 但目录名必须同步改成 consistent 版本

   例如：
   - `WORKDIR = ROOT / "clean_raw_hr_proxy_testset_100_consistent"`
   - `OUTDIR = ROOT / "chern_output_hr_proxy_testset_100_consistent"`
