# Jekyll 博客翻页逻辑更新总结





## 一、这次修改的核心目的

这次更新翻页逻辑，核心目标有四个：

1. 解决网站文章数量较多时，页面加载过慢的问题。
2. 将原先依赖前端 JavaScript 的“假分页”改成真正的静态分页。
3. 让各分类页（如 `life`、`reading`、`writing` 等）支持真正的分页访问。
4. 将 GitHub Pages 的部署方式从原先的默认分支构建，切换到 GitHub Actions 自定义构建，以便支持 `jekyll-paginate-v2` 和自动分类页生成。

---

## 二、原先的问题：前端假分页导致页面越来越卡

在原先的方案中，多个栏目页都采用了类似的逻辑：

```liquid
window.allPosts = [
    {% raw %}{% for post in site.posts %}{% endraw %}
```

或者：

```liquid
window.allPosts = [
    {% raw %}{% for post in site.categories.xxx %}{% endraw %}
```

再配合：

```html
<div id="posts-container"></div>
<div id="pagination" class="pagination">...</div>
<script src="{{ '/assets/vendor/startbootstrap-clean-blog/js/pagination.js' | relative_url }}"></script>
```

由浏览器前端一次性加载所有文章数据，然后通过 `pagination.js` 在浏览器中“假装分页”。

### 这种做法的问题

1. 页面一打开就要把大量文章数据全部塞进前端。
2. 分类页文章越多，页面体积越大，浏览器解析越慢。
3. 首页、分类页、搜索页都可能重复加载大量文章数据。
4. 读者点击“下一页”时并没有真的加载新页面，而只是前端切换已加载的数据。
5. 不利于后续维护，也不利于 GitHub Pages 上的稳定展示。

---

## 三、为什么要切换到真分页

目标是实现：

- 首页只显示当前页的文章
- 分类页只显示该分类当前页的文章
- 用户点“下一页”时，访问的是新的静态 HTML 页面
- 页面不再一次性加载所有文章数据

也就是说，要从：

**前端 JS 分页**

切换到：

**Jekyll 构建阶段生成的真分页**

---

## 四、为什么部署方式也要改

一开始网站是直接依赖 GitHub Pages 默认构建的。  
这种方式虽然简单，但有一个限制：

- GitHub Pages 原生受支持的插件有限
- `jekyll-paginate-v2` 不适合继续走默认受限构建

因此后续改成了：

- 本地修改文件
- 使用 GitHub Desktop commit / push
- GitHub Actions 自动构建和部署到 GitHub Pages

### 这个改动的意义

1. 平时的写作和 push 流程基本不变。
2. 构建逻辑可以自己控制。
3. 可以使用 `jekyll-paginate-v2`。
4. 可以自动生成分类页和分页页。
5. 更适合文章数量越来越多的网站。

## 五、这次涉及和修改过的主要文件

下面按功能分类总结。

---

## 1. 导航栏结构文件

**文件：**

```text
_includes/navbar.html
```

### 作用

- 控制导航栏链接
- 决定各分类页入口路径

### 与本次翻页逻辑相关的点

导航栏中的分类入口必须和自动生成分类页路径保持一致，例如：

```html
<a class="nav-link" href="{{ '/writing' | relative_url }}">Writing</a>
```

对应的分类自动页也应该生成到：

```text
/writing/
```

### 注意事项

- 导航栏路径建议统一使用小写
- 各分类名也建议统一使用小写
- 避免出现 `/Writing/` 和 `/writing/` 混用的情况

---

## 2. 原先的前端分页脚本

**文件：**

```text
assets/vendor/startbootstrap-clean-blog/js/pagination.js
```

### 原作用

- 接收 `window.allPosts`
- 在浏览器前端进行分页切换

### 本次更新后的处理

这个文件不再作为分类页和首页分页的核心逻辑使用。  
真分页完成后：

- 分类页不再依赖 `window.allPosts`
- 首页不再依赖 `window.allPosts`
- `pagination.js` 逻辑退出主要舞台

### 结论

这个脚本代表的是**旧的前端假分页方案**。

---

## 3. 站点配置文件

**文件：**

```text
_config.yml
```

### 这是本次更新最关键的文件之一

最终改成了支持：

- `jekyll-paginate-v2`
- `autopages`
- 分类自动分页页

### 最终核心配置逻辑

```yaml
plugins:
  - jekyll-feed
  - jekyll-paginate-v2
  - jekyll-sitemap

pagination:
  enabled: true
  per_page: 5
  permalink: "/page:num/"
  sort_reverse: true

autopages:
  enabled: true
  categories:
    enabled: true
    layouts:
      - autopage_category.html
    title: ":cat"
    permalink: "/:cat/"
    slugify:
      mode: "default"
      case: false
  tags:
    enabled: false
  collections:
    enabled: false
```

### 这个文件里曾经遇到的问题

#### 问题 1：分页功能被关闭
曾经写成：

```yaml
pagination:
  enabled: false
```

这会导致：

- 自动分类页存在
- 但 `paginator.posts` 不可用
- 页面看起来生成了，但文章列表为空

#### 解决方法

改为：

```yaml
pagination:
  enabled: true
```

并补充 `per_page`、`permalink` 等分页参数。

---

#### 问题 2：分类 slug 大小写不一致
在自动生成分类页时，如果分类名或 slug 大小写不统一，GitHub Pages 上容易出现：

- 本地正常
- 线上 404

#### 解决方法

增加：

```yaml
slugify:
  mode: "default"
  case: false
```

强制分类页路径统一小写。

---

## 4. 页面通用布局文件

**文件：**

```text
_layouts/page.html
```

### 原问题

原先这个文件里仍然内嵌了旧的前端假分页逻辑，例如：

- `posts-container`
- `pagination`
- `window.allPosts`
- `pagination.js`

这会导致所有 `layout: page` 页面都可能被自动注入旧分页数据。

### 解决方法

将 `_layouts/page.html` 改成真正的“纯页面布局”：

- 只负责页头显示
- 只负责 `{{ content }}`
- 不再注入文章列表和前端分页逻辑

### 最终作用

- 普通页面（About、Search 等）正常显示正文
- 不再意外地加载大量文章数据

---

## 5. 首页布局文件

**文件：**

```text
_layouts/home.html
```

### 原问题

About 等页面曾经使用：

```yaml
layout: home
```

但 `home` 布局后来被改成了首页专用的文章分页布局：

- 显示站点标题
- 遍历 `paginator.posts`
- 不输出普通页面正文

这会导致：

- About 页面显示成 Home 的样子
- 但没有正确显示正文

### 解决方法

- `layout: home` 只留给首页用
- About / Search / 普通说明页改成 `layout: page`

### 结论

`home` 只服务首页真分页，不再给普通正文页面使用。

---

## 6. 自动分类页布局文件

**文件：**

```text
_layouts/autopage_category.html
```

### 这是分类页真分页的核心模板

自动生成的分类页（如 `/life/`、`/reading/`、`/writing/`）都依赖这个布局。

### 主要职责

1. 显示分类页头部
2. 根据分类名选择背景图
3. 根据分类名显示合适标题
4. 遍历 `paginator.posts`
5. 输出上一页 / 下一页按钮

### 最终版模板解决了这些问题

#### 问题 1：标题显示为小写
例如页面标题显示为：

```text
life
```

而不是：

```text
Life
```

#### 解决方法

在模板里手动为分类名映射标题：

- `life` -> `Life`
- `sci-log` -> `Sci-Log`
- `sci-note` -> `Sci-Note`
- `reading` -> `Reading`
- `learning` -> `Learning`
- `writing` -> `Writing`

---

#### 问题 2：页面多出“life 分类文章”之类的说明文字
原先模板中有类似：

```liquid
<p>{{ cat_name }} 分类文章</p>
```

#### 解决方法

直接删除这行，不再显示多余说明。

---

#### 问题 3：分类页背景图没有正确载入
因为自动分类页不会自动继承原手写页面中的：

```yaml
background: ...
```

#### 解决方法

在 `_layouts/autopage_category.html` 中按分类名手动分配背景图：

- `life` -> `/img/bg-life.jpg`
- `sci-log` -> `/img/bg-sci-log.jpg`
- `sci-note` -> `/img/bg-note.jpg`
- `reading` -> `/img/bg-reading.jpg`
- `learning` -> `/img/bg-about.jpg`
- `writing` -> `/img/bg-travel.jpg`

## 7. GitHub Actions 工作流文件

**文件：**

```text
.github/workflows/pages.yml
```

### 作用

- 自动构建站点
- 自动部署到 GitHub Pages

### 最终工作流作用

1. checkout 仓库
2. 配置 Pages
3. 配置 Ruby
4. `bundle install`
5. `bundle exec jekyll build --trace`
6. 上传 `_site`
7. 部署到 `github-pages` environment

### 本次遇到的关键问题

#### 问题 1：workflow 不自动触发
最开始 workflow 写的是：

```yaml
on:
  push:
    branches: ["main"]
```

但实际 push 的分支是：

```text
master
```

所以：

- 不会自动触发
- 只能手动点击 `Run workflow`

#### 解决方法

改成：

```yaml
on:
  push:
    branches: ["master", "main"]
  workflow_dispatch:
```

这样：

- push 到 `master` 会自动部署
- push 到 `main` 也能自动部署
- 仍保留手动运行能力

---

#### 问题 2：Actions 页面只看到旧的 `pages-build-deployment`
这通常说明：

- 自己的 workflow 没被 GitHub 识别
- 文件位置错误
- Pages Source 没切到 GitHub Actions

#### 解决方法

确认：

```text
.github/workflows/pages.yml
```

存在并已 push；同时在：

```text
Settings -> Pages -> Build and deployment -> Source
```

中切换到：

```text
GitHub Actions
```

原先存在一些手写分类页，例如：

```text
life.md
reading.md
sci-log.md
sci-note.md
learning.md
writing.md
```

### 原作用

### 新问题

- `/life/`

### 处理建议

## 六、这次迁移过程中遇到的主要问题与解决办法

下面按问题类型汇总。

---

## 问题 1：首页与分类页都在加载大量文章，导致站点越来越卡

### 原因

前端假分页方案会把大量文章数据写入页面中。

### 解决方法

采用 Jekyll 构建时真分页：

- 首页走 `home` 布局的 `paginator.posts`
- 分类页走 `autopage_category.html` 的 `paginator.posts`

---

## 问题 2：本地能用，GitHub 上分类页 404

### 原因之一

workflow 最初只监听 `main`，实际 push 到 `master`，自动部署没有正常触发。

### 解决方法

将 workflow 的分支监听改成：

```yaml
branches: ["master", "main"]
```

---

### 原因之二

未来日期文章在 GitHub 构建时被排除。

Jekyll 默认不发布未来文章。如果某个分类（如 `writing`）目前只有未来日期文章，那么：

- 单篇文章不会生成
- 分类首页也不会生成

### 解决方法

将文章日期改到当前时间之前，或显式调整 `date` 字段。  
例如将：

- 文件名日期从未来日期改成当前或过去日期
- front matter 中明确加 `date`

---

## 问题 3：本地构建和远端部署不一致

### 表现

- 本地分类页正常
- 远端访问 404
- artifact 中缺少分类目录

### 解决方法

从 Actions build 日志和 artifact 反查：

1. 检查日志中是否有：
   - `AutoPages:`
   - `Pagination:`
2. 下载 artifact，看是否生成：
   - `writing/index.html`
   - `writing/page2/index.html`
   - 单篇文章页面

通过这个过程可以判断问题是出在：

- 构建阶段
- 还是发布阶段

---

## 问题 4：About 页面显示成 Home 的样子

### 原因

About 页面使用了：

```yaml
layout: home
```

而 `home` 布局已经被改成首页专用文章分页布局。

### 解决方法

About 页改成：

```yaml
layout: page
permalink: /about/
```

---

## 问题 5：自动分类页背景图丢失、标题小写、多余说明文字

### 原因

自动生成分类页不再读取原手写页面的 front matter。

### 解决方法

在 `_layouts/autopage_category.html` 中：

- 手动映射标题
- 手动映射背景图
- 删除多余说明文字

---

## 问题 6：workflow 存在 warning

### 表现

GitHub Actions 中出现 Node.js 20 deprecation warning。

### 结论

这类 warning 当前不是导致部署失败或分类页 404 的直接原因。  
只要 `build` 和 `deploy` 绿了，当前部署仍然是成功的。

### 处理建议

后续有空时再统一升级 action 版本。

---

## 问题 7：本地测试 Ruby / Bundler 环境不完整

### 表现

最开始执行：

```bat
bundle install
```

提示：

```text
'bundle' 不是内部或外部命令
```

### 解决方法

1. 安装 RubyInstaller for Windows
2. 勾选加入 PATH
3. 安装 MSYS2-Devkit
4. 执行：

```bat
gem install bundler jekyll
```

5. 再运行：

```bat
bundle install
bundle exec jekyll serve
```

## 七、本次最终形成的翻页逻辑结构

现在网站的翻页逻辑，目标结构如下：

### 首页
- 使用 `layout: home`
- 使用 `paginator.posts`
- 真分页 URL：
  - `/`
  - `/page2/`
  - `/page3/`

### 分类页
- 由 `jekyll-paginate-v2 + autopages` 自动生成
- 使用 `_layouts/autopage_category.html`
- 真分页 URL：
  - `/life/`
  - `/life/page2/`
  - `/reading/`
  - `/reading/page2/`
  - `/writing/`
  - `/writing/page2/`

### 普通页面
- 使用 `layout: page`
- 只显示 `{{ content }}`

## 八、当前分类建议

目前建议作为自动分类真分页的分类：

- `life`
- `sci-log`
- `sci-note`
- `reading`
- `learning`
- `writing`

不建议作为自动分类页处理的页面：

- `search`
- `about`

因为它们更适合作为普通页面，而不是文章分类归档页。

## 九、后续维护建议

### 1. 分类名统一小写
文章 front matter 中统一写：

```yaml
categories: writing
categories: life
categories: reading
categories: learning
categories: sci-log
categories: sci-note
```

不要混用：

- `Writing`
- `WRITING`
- `writing`

---

### 2. 文章 permalink 尽量统一带尾斜杠
例如：

```yaml
permalink: /writing/20260327strange-things/
```

这样更符合静态站点目录结构，也更稳定。

---

### 3. 避免未来日期文章直接影响线上分类页
如果某分类只有未来文章，那么线上分类页可能不会生成。

建议：

- 发布前确认日期不是未来时间
- 或谨慎使用 `future: true`

---

### 4. 删除旧的手写分类页残留
一旦确认自动分类页稳定运行，应删除旧的：

- `life.md`
- `reading.md`
- `sci-log.md`
- `sci-note.md`
- `learning.md`
- `writing.md`

避免路径冲突和维护混乱。

---

### 5. Action warning 后续再统一处理
当前可以先不影响正常写作与部署。  
后续若要升级，可统一更新 workflow 中的 action 版本。

## 十、最终总结

这次翻页逻辑更新，本质上是一次从：

**前端假分页 + GitHub Pages 默认构建**

迁移到：

**Jekyll 构建时真分页 + GitHub Actions 自动部署**

的系统性重构。

### 这次重构解决了什么

1. 减少页面一次性加载全部文章带来的卡顿。
2. 首页、分类页都转向真正的静态分页。
3. 分类页支持自动生成和自动分页。
4. 文章数量增多后站点结构更清晰。
5. 本地与线上部署链路更加明确。

### 这次更新最重要的经验

1. 先分清“构建问题”和“发布问题”。
2. 本地正常不代表远端部署一定一致。
3. GitHub Actions 的分支监听必须和实际分支一致。
4. 自动生成分类页时，要避免旧手写页抢同一路径。
5. 未来日期文章会直接影响线上分类页是否生成。

### 当前状态

翻页逻辑已经完成从旧方案向新方案的迁移，整体方向是正确的。  
后续只需继续做少量清理和细节统一，就可以形成一个更稳定、更适合长期维护的博客结构。
